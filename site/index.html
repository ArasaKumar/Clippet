Why Clippet

Windows 11's built-in clipboard history (Win+V) has no search box — once your history grows past a screenful, finding the snippet you copied an hour ago means scrolling through every item by hand. Clippet adds fuzzy search across every entry, unlimited pinned items, and high-fidelity paste-back for RTF / HTML / images / files, all in a single binary you can drop anywhere.

Built for the way you actually work

Captures the format that matters

Clippet inspects every clipboard write and stores the most informative format available — not a flattened text version of it. Files paste back as files, RTF keeps its styling, and images show a thumbnail inline and round-trip through PNG.

Lives where you do

A global Ctrl + Shift + V hotkey summons the popup at your cursor. Pick an item with arrow keys + Enter. Clippet restores focus to the previous window and synthesizes a real Ctrl+V — the paste lands exactly where it would have.

Fuzzy search built in

Just start typing. Skim-style fuzzy matching across every item's preview, with matched characters highlighted in bold and rows reordered by score. Pinned items still float to the top.

Pin anything important

Star an item to pin it. Pinned entries sort above the rest, survive the 200-item history cap forever, and stay across restarts. Right-click any row for Paste / Pin / Copy / Delete.

Looks like Windows 11

Acrylic backdrop, rounded corners via DWM, custom flat title bar, per-monitor DPI v2, light/dark palette tracked from the system theme. Segoe UI Variable for body, semibold heading on the title bar.

Resilient by design

Atomic disk writes (write to .tmp then rename) — no half-written files if power dies mid-save. Image payloads live as standalone PNGs alongside history.json, so the metadata file stays small and resident RAM stays bounded no matter how big a screenshot you copy. Every Win32 call has paired cleanup. Pinned items are exempt from history overflow eviction.

How it works

Single-threaded by design

The Win32 message loop owns everything. There is no async runtime, no worker pool, no Mutex — just thread-local state read and written from the main thread. Less code, fewer bugs, lower memory.

Capture priority

On every WM_CLIPBOARDUPDATE, Clippet picks the richest available format:

Files → Spreadsheet → RTF → HTML → Image → Unicode text → ANSI text

Custom Win11 chrome

The popup is a WS_POPUP | WS_THICKFRAME window with the native caption suppressed. WM_NCHITTEST maps the outer 6 px to resize cursors and the top strip to a draggable region. DWM paints the rounded corners and acrylic backdrop.

No network code. Anywhere.

Clippet has zero network calls. No analytics SDK, no auto-update check, no remote logging, no cloud sync. Every byte of your clipboard data lives on your disk and nowhere else. The full source — about a thousand lines of Rust — is public domain. Audit it.

Build it yourself in two commands

No installer, no admin elevation, no Visual Studio project. Just cargo build.

Requirements

  • Windows 10 (build 1903+) or Windows 11
  • Rust toolchain via rustup
  • MSVC target x86_64-pc-windows-msvc (rustup default)

Build

PS> git clone https://github.com/ArasaKumar/Clippet.git
PS> cd Clippet
PS> cargo build --release
# target\release\clippet.exe is self-contained
PS> .\target\release\clippet.exe

FAQ

Why not just use Win+V?

The biggest gap is search. Windows' built-in clipboard history has no search box — once you've copied a few dozen things, finding a specific snippet means scrolling through every item by hand. It also doesn't tag entries by format, doesn't let pinned items survive the history cap, and can't be summoned by any hotkey other than Win+V. Clippet keeps a 200-item history, fuzzy-matches as you type, pins anything you mark indefinitely, and shows what each item is (text, RTF, HTML, image, file, spreadsheet, code) before you paste it.

Why Ctrl+Shift+V instead of Win+V?

The Windows 11 shell holds Win+V even when clipboard history is disabled in Settings, so it can't be reliably registered by another app. Ctrl+Shift+V is unclaimed in stock Windows and free across most apps.

Does Clippet conflict with Windows' own clipboard history?

No — they run independently and most users keep both. Win+V still gives you the last few items at a glance; Ctrl+Shift+V opens Clippet's full searchable history. If you'd rather have only one, disable the built-in one under Settings → System → Clipboard.

Where are images stored?

Re-encoded to PNG and written as individual files under %APPDATA%\Clippet\media\ — one full-resolution {id}.png per copy plus a {id}_thumb.png thumbnail for the listbox. Only filenames and dimensions live in history.json, so a 20 MB screenshot adds a few hundred bytes to the metadata file rather than ~26 MB of base64. Image bytes are streamed from disk on paste, kept off the resident heap until needed. Deleting a row removes both files; pinned images are exempt from the 200-item cap.

Why Rust + raw Win32 instead of a framework?

The whole point is the sub-MB binary and a low-double-digit-MB idle footprint (~15 MB working set, < 2 MB private bytes). A framework would add tens of MB of runtime for features Clippet doesn't need.

Can I trust this with sensitive data?

The same trust you give any local app that reads your clipboard. Clippet writes to a per-user AppData file that other users on the machine can't read. Source is public domain — audit it.

How do I clear history?

Right-click the tray icon → Clear history. Or delete %APPDATA%\Clippet\history.json while Clippet isn't running.
